v0.3.82026-03-10

Runtime limits, examples manifest hardening & WASM-ready core bootstrap

  • Added settings dialog Settings > Uso máximo de Memória... to configure two runtime caps in a single prompt: memory budget (GB) and max backsteps.
  • New persisted preferences: maxMemoryGb (default 2) and maxBacksteps (default 100), wired directly into engine settings.
  • Engine memory writes now enforce maxMemoryBytes; over-limit allocations raise a controlled runtime error instead of runaway growth.
  • Backstep history pruning now obeys dynamic limits during preference changes and supports zero-history mode.
  • Example loading now trusts only examples/examples.json; removed server-dependent directory autodiscovery to guarantee deterministic menus on all hosts.
  • Switched runtime engine bootstrap to createMarsEngine() factory (JS fallback by default, WASM selectable via backend setting).
  • Introduced 00-core-wasm-bridge.js with window.WebMarsWasmCore API (init, registerFactory, createEngineSync) to support future C++/WebAssembly core migration without refactoring UI/runtime modules.
  • Added no-interaction rendering mode in max speed: UI highlights and auto-scroll are disabled while running at “Run speed at max (no interaction)” to reduce layout thrash and keep the browser responsive.
  • Run loop tuning for responsiveness: smaller unlimited batches, tighter cooperative time budget, and throttled UI sync in non-interactive mode.
  • Initial C++ hotpath module scaffold added (web/wasm/core_hotpath.cpp) plus browser loader 00-core-wasm-hotpath.js and build script node wasm/build-wasm-core.mjs (Emscripten).
  • Updated bootstrap module order in app.bundle.js to load WASM bridge before UI/runtime.
