Escrevendo e usando macros

.macro, .end_macro,.eqv e .incluir diretivas são novas no MARS 4.3

Introdução às macros

Patterson e Hennessy definem uma macro como uma instalação de correspondência e substituição de padrões que fornece um mecanismo simples para nomear uma sequência de instruções frequentemente utilizada [1]. Isso permite que o programador especifique a sequência de instruções invocando a macro. Isso requer apenas uma linha de código para cada uso em vez de digitar repetidamente na sequência de instruções de cada vez. Segue-se o axioma "defina uma vez, use muitas vezes", que não só reduz a chance de erro, mas também facilita a manutenção do programa.

As macros são como procedimentos (sub-rotinas) neste sentido, mas operam de forma diferente dos procedimentos. Procedimentos em linguagem de montagem MIPS seguem protocolos específicos para definição de procedimento, chamada e retorno. As macros operam substituindo o corpo macro para cada uso no momento da montagem. Essa substituição é chamada de expansão macro.. Eles não exigem os protocolos e a execução de despesas gerais dos procedimentos.

Como um exemplo simples, você pode querer encerrar seu programa a partir de vários locais. Se você estiver correndo do MARS IDE, você usará a chamada do sistema 10, saída. A sequência de instruções é muito fácil

		li $v0,10
		syscall
mas ainda tedioso. Você pode definir uma macro, vamos chamá-la feito, para representar esta sequência 
		.macro feito
		li $v0,10
		syscall
		.end_macro
em seguida, invocá-lo sempre que desejar com a declaração 
		feito
No momento da montagem, o montador substituirá cada ocorrência da declaração feito com a declaração de duas sequência 
		li $v0,10
		syscall
Esta é a expansão macro. O simulador de tempo de execução desconhece macros ou expansão de macro.

Se estiver executando o MARS a partir da linha de comando, talvez você queira retornar um valor de terminação. Isso pode ser feito com syscall 17, exit2, que toma o valor de rescisão como argumento. Uma macro equivalente, Vamos chamar isso terminar seria 

		.macro termina (%termination_value)
		li $a0, %termination_value
		li $v0, 17 anos
		syscall
		.end_macro
Esta macro define a parâmetro formal para representar o valor de rescisão. Você o invocaria com a declaração 
		terminar (1)
para terminar com o valor 1. Após a montagem, a declaração terminar (1) seria substituído pela sequência de três declarações 
		li $a0, 1
		li $v0, 17 anos
		syscall
O valor do argumento, 1, é substituído onde quer que o parâmetro formal %termination_value aparece no corpo macro. Trata-se de uma substituição textual. Observe que neste exemplo o valor do argumento deve ser um número inteiro, não um registrar nome ou uma etiqueta, porque o parâmetro é usado como o segundo operando na operação Load Immediate.

No MARS, uma macro é semelhante a uma instrução estendida (pseudo). distinguem-se na medida em que a expansão de instruções estendidas é suportado por uma linguagem de especificação e mecanismo definido internamente que pode manipular valores de argumento. A facilidade macro só pode substituir os valores de argumento conforme dados, e usa um mecanismo separado de instruções estendidas.

Seguem-se exemplos e detalhes adicionais.

Como definir macros

A primeira linha começa com a .macro directiva seguida de uma lista facultativa de parâmetros formais. Colocar vírgulas entre parâmetros e parênteses ao redor da lista é opcional.

Cada parâmetro formal é um identificador que começa com um % caráter. Para compatibilidade com o APP pré-processador SPIM, ele pode, alternativamente, começar com $.

As linhas que se seguem definem o corpo da macro. Use os parâmetros formais conforme apropriado. O corpo pode conter segmentos de dados, bem como segmentos de texto.

A definição macro termina com a .end_macro diretiva.

Veja as Notas abaixo para obter informações adicionais.

Como usar macros

Para invocar uma macro, forme uma instrução que consiste no nome da macro e, em seguida, um token para cada argumento a ser substituído pelo seu parâmetro formal correspondente pelo montador. A lista de argumentos pode, opcionalmente, ser cercada por parênteses. Os argumentos podem ser separados por espaços ou vírgulas.

A expansão macro é uma tarefa de pré-processamento para montadores.

Notas

  • Uma definição macro deve aparecer antes de sua utilização. Sem referências para a frente.
  • Todas as definições de macro são locais em cada arquivo e não podem ser globais.
  • Definições macro aninhadas não são suportadas. Não .macro directiva deve aparecer dentro de um corpo de uma definição macro.
  • Uma definição de macro pode conter uma chamada para uma macro previamente definida. Apenas referências atrasadas são permitidas.
  • Os rótulos colocados no corpo de uma definição macro não terão o mesmo nome após a expansão da macro. Durante a expansão, seu nome será seguido por "_M#" onde # será um número único para cada expansão macro.
  • Duas macros com o mesmo nome, mas um número diferente de parâmetros são considerados diferentes e ambos podem ser usados.
  • Uma macro definida com o mesmo nome e mesmo número de parâmetros como outra macro definida antes de ser ignorada.
  • Cada argumento em uma chamada de macro só pode ser um único elemento de idioma (token). Por exemplo "4($t0)" não pode ser um argumento.
  • As macros fazem parte do montador, não do ISA. Portanto, a sintaxe pode ser diferente com outros montadores. Para compatibilidade com o simulador SPIM, as macros estilo SPIM também são suportadas no MARS. As macros estilo SPIM são iguais ao MARTE, mas os parâmetros formais começam com "$" em vez de "%".

Exemplos

  • Imprimir um número inteiro (o argumento pode ser um valor imediato ou um nome de registo): 
    	.macro print_int (%x)
	li $v0, 1
	adicionar $a0, $zero, %x
	syscall
	.end_macro
	
	print_int ($s0)
	print_int (10)
  • Imprimir uma string (a macro primeiro atribuirá um rótulo ao seu parâmetro no segmento de dados e depois o imprimirá): 
    	.macro print_str (%str)
	.data
myLabel: .asciiz %str
	.texto
	li $v0, 4
	la $a0, myLabel Em inglês
	syscall
	.end_macro
	
	print_str ("test1") #"test1" será rotulado com o nome "myLabel_M0"
	print_str ("test2") #"test2" será rotulado com o nome "myLabel_M1"
  • Implementando um for-loop simples: 
    	# mecanismo de looping genérico
	.macro para (%regiterator, %from, %to, %bodyMacroName)
	adicionar %regiterator, $zero, %from
	Loop:
	%bodyMacroName ()Tradução
	adicionar %regiterator, %regiterator, 1
	ble %regiterator, %to, Loop
	.end_macro
	
	#imprimir um número inteiro
	corpo .macro()
	print_int $t0
	print_str "\n"
	.end_macro
	
	#Impressão de 1 a 10:
	para ($t0, 1, 10, corpo)
    O para macro tem 4 parâmetros. %regiterator deve ser o nome de um registo que itera de %de para %to e em cada iteração %bodyMacroNome será ampliada e executada. Argumentos para %de e %to pode ser um nome de registro ou um valor imediato, e %bodyMacroNome deve ser nome de uma macro que não tem parâmetros.

Números de linha macro source

Para fins de mensagens de erro e exibição de segmento de texto, o MARS tenta exibir números de linha para a definição e o uso do pertinente declaração macro. Se uma mensagem de erro mostrar o número da linha no formulário "X->Y" (ex. "20->4"), então X é o número da linha na expansão (utilização) onde o erro foi detectado e Y é o número da linha na definição macro. Na exibição Segmento de texto de código fonte, a definição macro número de linha será exibido dentro de colchetes, por exemplo. "<4>", no ponto de expansão. Números de linha devem corresponder ao números que você veria no editor de texto.

A directiva .eqv

O .eqv diretiva (abreviação de "equivalência") é também nova no MARS 4.3. É semelhante a #defina em C ou C++. É é usado para substituir uma string arbitrária por um identificador. É útil, mas muito menos poderoso que macros. Foi desenvolvido independentemente da instalação macro.

Usando .eqv, você pode especificar substituições simples que fornecem "definir uma vez, usar muitas vezes" capacidade na montagem tempo de pré-processamento. Por exemplo, uma vez que você define 

       .eqv  LIMIT      20
       .eqv  CTR        $t2
       .eqv  CLEAR_CTR  add  CTR, $zero, 0
então você pode se referir a eles no código subsequente: 
       li $v0,1
       CLEAR_CTR
loop:  move $a0, CTR
       syscall
       add   CTR, CTR, 1
       blt   CTR, LIMIT, loop
       CLEAR_CTR
Durante o pré-processamento de montagem, o .eqv substituições serão aplicadas. O código resultante é 
       li   $v0,1
       add  $t2, $zero, 0
loop:  move $a0, $t2
       syscall
       add  $t2, $t2, 1
       blt  $t2, 20, loop
       add  $t2, $zero, 0
que quando executado irá exibir os valores de 0 a 19 em uma linha sem espaços intervenientes.

Note that the substitution string is not limited to a single token. Like .macro, .eqv is local to the file in which it is defined, and must be defined prior to use. Macro bodies can contain references to .eqv directives.

The .include directive

The .include directive is also new in MARS 4.3. It has one operand, a quoted filename. When the directive is carried out, the contents of the specified file are substituted for the directive. This occurs during assembly preprocessing. It is like #include in C or C++.

.include is designed to make macro and equivalence (.eqv directive) use more convenient. Both macro definitions and equivalence definitions are local, which means they can be used only in the same file where defined. Without .include, you would have to repeat their definitions in every file where you want to use them. Besides being tedious, this is poor programming practice; remember "define once, use many times." Now you can define macros and equivalences in a separate file, then include it in any file where you want to use them.

The .include preprocessor will detect and flag any circular includes (file that includes itself, directly or indirectly).

The use of .include presents some challenges for error messaging and for source code numbering in the Text Segment display. If a file being included has any assembly errors, the filename and line number in the error message should refer to the file being included, not the file it was substituted into. Similarly, the line number given in the Text Segment source code display refers to the line in the file being included. Thus the displayed line numbers do not monotonically increase - this is also the case when using the "assemble all" setting. Line numbers should correspond to the numbers you would see in the text editor.

As a simple example, you could define the done macro (and others) in a separate file then include it wherever you need it. Suppose "macros.asm" contains the following: 

		.macro done
		li $v0,10
		syscall
		.end_macro
Você pode incluí-lo em um arquivo de origem diferente, algo assim: 
		.include "macros.asm"
		.data
value:		.word	 13
		.text
		li	 $v0, 1
		lw	 $a0, value
		syscall
		done

During assembly preprocessing, this would be expanded to 

		.macro done
		li $v0,10
		syscall
		.end_macro	
		.data
value:		.word	 13
		.text
		li	 $v0, 1
		lw	 $a0, value
		syscall
		done

The assembler will then perform the appropriate macro expansion.
