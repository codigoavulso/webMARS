MARTE - Simulador de montagem e tempo de execução de microfones

Lançamento 4.5

Agosto de 2014

Requisitos de operação

O MARTE está escrito em Java e requer pelo menos a liberação 1.5 do J2SE Java Runtime Environment (JRE) para funcionar. O IDE gráfico é implementado usando o Swing. Tem sido testado no Windows XP, Vista e 7; Mac OS X; e também está sendo usado no Linux.

Algumas limitações de montador e simulador de MARTE

Libera 3.0 e depois monta e simula quase todas as instruções MIPS32 documentadas no livro didático Organização e Design de Computadores, Quarta Edição por Patterson e Hennessy, Elsevier - Morgan Kaufmann, 2009. Todas as instruções básicas e pseudo, directivas, e serviços de sistema descritos no Apêndice B são implementados.

As limitações do MARS a partir do Release 4.5 incluem:

  • Segmentos de memória (texto, dados, pilha, texto do kernel, dados do kernel) são limitados a 4MB cada um começando em seu respectivos endereços base.
  • Não há modo pipelined (mas a ramificação atrasada é suportada).
  • Se você abrir um arquivo que é um link ou atalho para outro arquivo, o MARS não abrir o arquivo de destino. A caixa de diálogo aberta do arquivo é implementada usando o JFileChooser do Java Swing, que não suporta links.
  • Muito poucas alterações de configuração, além daquelas no menu Configurações, são salvas de uma sessão para a próxima. As configurações do editor, que incluem configurações de fonte e exibição de números de linha, são salvas.
  • O IDE funcionará apenas com o montador do MARS. Não pode ser usado com qualquer outro compilador, montador ou simulador. O montador e simulador MARS pode ser usado através do IDE ou de um prompt de comando.
  • Bug: O iluminador de mensagens de erro não seleciona automaticamente o código para o primeiro conjunto erro se o ficheiro que contém o erro não estiver aberto no momento da montagem (montar-em-aberto, montar-tudo).
  • Bug: O recurso de captura de tela do Screen Magnifier não parece funcionar corretamente no Windows Vista.
  • Bug: Parece haver um vazamento de memória no Editor. Várias pessoas diferentes têm de forma independente relatou o mesmo comportamento: desaceleração severa na resposta do editor durante uma sessão interativa estendida. Se o MARS for excluído e reiniciado, esse comportamento desaparece e o editor responde instantaneamente às ações.
  • Não é um bug, mas documentado aqui de qualquer maneira: os endereços de destino de instruções do MIPS Branch são representados por o número relativo de palavras para ramificar. Com o Release 3.4, esse valor reflete ramificação atrasada, independentemente de a configuração de Ramificação Atrasada estar ativada ou não. O código binário gerado para ramificações agora corresponderá ao de exemplos na Organização de Computadores e livro didático de design. Esta é uma mudança em relação ao passado, e foi feita após extensas discussões com vários docentes adotantes da MARS. Anteriormente, o deslocamento de ramificação era 1 menor se a configuração de ramificação retardada foi habilitado -- a instrução etiqueta: beq $0, $0, label geraria 0x1000FFFF se Ramificação atrasada foi habilitada e 0x100000000 se estivesse desativado. Agora vai gerar 0x1000FFFF em qualquer caso. O simulador sempre se ramificará até o local correto; o MARTE não permite a montagem sob um configuração e simulação sob o outro para ocorrer.
