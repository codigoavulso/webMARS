Funções SYSCALL disponíveis no MARS

Introdução

Uma série de serviços do sistema, principalmente para entrada e saída, estão disponíveis para uso por seu programa MIPS. São descritos na tabela abaixo.

O conteúdo do registro MIPS não é afetado por uma chamada do sistema, exceto para registros de resultados conforme especificado na tabela abaixo.

Como usar os serviços do sistema SYSCALL

Passo 1. Carregue o número de serviço no registro $v0.
Passo 2. Carregue valores de argumento, se houver, em $a0, $a1, $a2, ou $f12 conforme especificado.
Passo 3. Emitir a instrução SYSCALL.
Passo 4. Recuperar valores de retorno, se houver, de registros de resultados conforme especificado.

Exemplo: exibir o valor armazenado em $t0 no console
    li  $v0, 1           # service 1 is print integer
    add $a0, $t0, $zero  # load desired value into argument register $a0, using pseudo-op
    syscall

NOTAS: Serviços numerados com 30 anos ou superior não são fornecidos pela SPIM
Serviço 8 - Segue a semântica de UNIX 'fgets'. Para o comprimento especificado n, a corda não pode ser mais do que n-1. Se menos do que isso, adiciona newline ao fim. Em ambos os casos, então pads com byte nulo Se n = 1, entrada é ignorada e byte nulo colocado no endereço de buffer. Se n < 1, entrada é ignorada e nada é escrito para o buffer.
Serviço 11 - Imprime o caractere ASCII correspondente ao conteúdo de byte de ordem baixa.
Serviço 13 - MARS implementa três valores de sinalizador: 0 para somente leitura, 1 para write-only com create, e 9 para write-only com create e append. Ele ignora o modo. O descritor de arquivo retornado será negativo se a operação falhar. O arquivo subjacente I/O implementação utiliza java.io.FileInputStream.read() para ler e java.io.FileOutputStream.write() para escrever. O MARS mantém os descritores de arquivo internamente e os aloca a partir de 3. Os descritores de arquivo 0, 1 e 2 são sempre aberto para: leitura de entrada padrão, escrita para saída padrão e escrita para erro padrão, respectivamente (novo na versão 4.3).
Serviços 13,14,15 - Em MARS 3.7, o registro de resultados foi alterado para $v0 para compatibilidade SPIM. Anteriormente $a0 foi impresso erroneamente no Apêndice B de Organização e Design de Computadores,.
Serviço 17 - Se o programa MIPS for executado sob controle da interface gráfica MARS (GUI), o código de saída em $a0 é ignorado.
Serviço 30 - O tempo do sistema vem de java.util.Date.getTime() como milissegundos desde 1 de Janeiro de 1970.
Serviços 31,33 - Simule a saída MIDI através de placa de som. Detalhes abaixo.
Serviços 40-44 usar geradores de número pseudoaleatório Java subjacentes fornecidos pelo java.util.Random classe. Cada fluxo (identificado por $a0 conteúdo) é modelado por um diferente Random objeto. Não há valores de sementes padrão, portanto, use o serviço Set Seed (40) se sequências aleatórias replicadas são desejadas.

Exemplo de arquivo I/O

O programa MIPS de exemplo abaixo abrirá um novo arquivo para escrever, escreverá texto a partir de um buffer de memória e, em seguida, o fechará. O arquivo será criado no diretório em que o MARS foi executado. 

# Sample MIPS program that writes to a new file.
#   by Kenneth Vollmar and Pete Sanderson

        .data
fout:   .asciiz "testout.txt"      # filename for output
buffer: .asciiz "The quick brown fox jumps over the lazy dog."
        .text
  ###############################################################
  # Open (for writing) a file that does not exist
  li   $v0, 13       # system call for open file
  la   $a0, fout     # output file name
  li   $a1, 1        # Open for writing (flags are 0: read, 1: write)
  li   $a2, 0        # mode is ignored
  syscall            # open a file (file descriptor returned in $v0)
  move $s6, $v0      # save the file descriptor 
  ###############################################################
  # Write to file just opened
  li   $v0, 15       # system call for write to file
  move $a0, $s6      # file descriptor 
  la   $a1, buffer   # address of buffer from which to write
  li   $a2, 44       # hardcoded buffer length
  syscall            # write to file
  ###############################################################
  # Close the file 
  li   $v0, 16       # system call for close file
  move $a0, $s6      # file descriptor to close
  syscall            # close file
  ###############################################################

Usando os serviços do sistema SYSCALL 31 e 33: saída MIDI

Esses serviços de sistema são exclusivos do MARS e fornecem um meio de produzir som. A saída MIDI é simulado pela sua placa de som do sistema, e a simulação é fornecida pelo javax.sound.midi pacote.

O serviço 31 irá gerar o tom e retornar imediatamente. O serviço 33 irá gerar o tom então dormir pela duração do tom antes de retornar. Assim, combina essencialmente os serviços 31 e 32.

Este serviço requer quatro parâmetros da seguinte forma:

pitch ($a0)

  • Aceita um valor positivo de byte (0-127) que denota um pitch como faria ser representado no MIDI
  • Cada número é um semitone / meio passo na escala cromática.
  • 0 representa um C muito baixo e 127 representa um G muito alto (um padrão 88 key piano começa em 9-A e termina em 108-C).
  • Se o valor do parâmetro estiver fora desse intervalo, ele aplica um valor padrão 60 que é o mesmo que o meio C em um piano.
  • Do meio C, todos os outros arremessos na oitava são os seguintes:
  • 61 = C# ou Db
  • 62 = D
  • 63 = D# ou Eb
  • 64 = E ou Fb
    		•
  • 65 = E# ou F
  • 66 = F# ou Gb
  • 67 = G
  • 68 = G# ou Ab
    		•
  • 69 = A
  • 70 = A# ou Bb
  • 71 = B ou Cb
  • 72 = B# ou C
    		•
  • Para produzir esses toms em outras oitavas, adicione ou subtraia múltiplos de 12.

    		•

    duração em milissegundos ($a1)

  • Aceita um valor de número inteiro positivo que é o comprimento do tom em milissegundos.
  • Se o valor do parâmetro for negativo, ele aplica um valor padrão de um segundo (1000 milissegundos).

    		•

    instrumento ($a2)

  • Aceita um valor positivo de byte (0-127) que denota o MIDI Geral "Patch" costumava tocar o tom.
  • Se o parâmetro estiver fora desse intervalo, ele aplica um valor padrão 0 que é um Grand Piano Acústico.
  • General MIDI padroniza o número associado a cada instrumento possível (muitas vezes referido como números de mudança de programa), no entanto, ele faz não determinar como o tom vai soar. Isto é determinado pelo sintetizador que está produzindo o som. Assim, um Tuba (patch 58) em um computador pode soar diferente do mesmo patch em outro computador.
  • Os 128 adesivos disponíveis são divididos em famílias de instrumentos de 8:
    • 0-7 piano 64-71 Reed
    8-15 Percussão Cromática 72-79 Tubulação
    16-23 Órgão 80-87 Chumbo de sintetizador
    24-31 Guitarra 88-95 Pad de sintetizador
    32-39 Baixo 96-103 Efeitos de sintetizador
    40-47 Cordas 104-111 Étnica
    48-55 Conjunto 112-119 Percussão
    56-63 Latão 120-127 Efeitos sonoros
  • Observe que fora do Java, o General MIDI geralmente se refere a patches 1-128. Quando se refere a uma lista de patches MIDI gerais, 1 deve ser subtraído para jogar o patch correto. Para uma lista completa dos instrumentos MIDI gerais, ver www.midi.org/about-midi/gm/gm1sound.shtml. O mapa de chave de percussão do canal MIDI geral 10 não é relevante para o método toneGenerator porque sempre padrão para o canal MIDI 1.

    		•

    volume ($a3)

  • Aceita um valor positivo de byte (0-127) onde 127 é o mais alto e 0 está em silêncio. Este valor denota velocidade MIDI que se refere à inicial ataque do tom.
  • Se o valor do parâmetro estiver fora desse intervalo, ele aplica um valor padrão 100.
  • A velocidade MIDI mede o quão difícil uma nota (ou nota) mensagem é reproduzida, talvez em um controlador MIDI como um teclado. A maioria Os sintetizadores MIDI traduzirão isso em volume em uma escala logarítmica em que a diferença de amplitude diminui à medida que o valor da velocidade aumenta.
  • Observe que o valor de velocidade em sintetizadores mais sofisticados também pode afetar o timbre do tom (como a maioria dos instrumentos soam diferentes quando eles são tocados mais alto ou mais suave).
